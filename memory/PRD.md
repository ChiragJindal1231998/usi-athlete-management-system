# Athlete Management System (AMS) — PRD

## Original problem statement
High-fidelity, production-feel prototype of an enterprise SaaS — the Athlete Management System (AMS) inside the Unified Sports Interface (USI). React + Tailwind, in-memory mock data only (no backend, no DB, no auth). Nine modules. Centered around the seeded scenario of **Arjun Sharma (SPR-014)**, 22, 100m sprinter, who is rehabbing a grade 2 right-hamstring strain that the AI predicted 4 days early.

## Architecture
- React 19 + react-router-dom 7 + Tailwind + shadcn/ui
- State: React Context (`AppContext`) with seed data (`/data/seed.js`)
- Charts: Recharts · Animations: Framer Motion (subtle) · Toasts: sonner · Icons: lucide-react
- Font: **DM Sans** (more distinctive than Inter while still enterprise-clean)
- No backend, no database, no env keys required.

## User personas (8, role switcher in TopBar)
- Federation admin · Performance director · Coach · Physiotherapist · Sports scientist · Nutritionist · Operations team · Athlete (self-view). The role switcher drives a **fully role-differentiated Dashboard** (each persona lands on a distinct, job-specific view — see below) plus a role-aware header chip and the signed-in profile in the account menu (`STAFF[role]`).

## Access control & data scoping (RBAC)
All nine nav tabs stay **visible to every role** ("show all tabs, scope data only" — no route guards, no hidden modules). Differentiation happens *inside* each module along two orthogonal dimensions, both centralised in `frontend/src/lib/access.js` and consumed via `AppContext`:

- **Data scope** — `scopeAthletes(role, athletes)`: an athlete sees only their own record (`ROLE_IDENTITY.athlete → SPR-014`, Arjun); a coach sees only athletes they coach (`ROLE_IDENTITY.coach → Meera Iyer`, i.e. Sprint A/B); all other staff are federation-wide. Modules read `scopedAthletes` / `me` (the self athlete, or null for staff) rather than the raw `athletes` list, and surface a `<ScopeNote>` banner stating what's in view.
- **Capabilities** — `can(role, capability)`: write affordances are gated by an owning-role map (e.g. `injury.report`/`injury.advance` → physio, `training.acceptAI`/`training.moveSession` → coach, `nutrition.edit` → nutritionist, `assessment.record` → scientist/director, `athletes.add`/`athletes.verify` → ops/admin, `analytics.export` → ops/director/admin, `self.*` → athlete). Non-owning roles see the same data **read-only** (buttons hidden or replaced with status badges).
- **Athlete exception (explicit):** the athlete persona never sees the "add athlete" affordance and the Copilot snapshot is scoped to their own record only (`me` → stats/athletes/injuries filtered to self, alerts emptied), so the LLM can never answer about other athletes.

## Core requirements implemented
1. **Dashboard / Command center (role-differentiated)** — `Dashboard.jsx` is a thin router that dispatches by `role` to one of eight purpose-built views under `components/dashboard/` (each takes `onOpenAthlete` and reuses StatCard / Card / AIInsight / StatusBadge / shared widgets), with a role-aware `PageHeader` surfacing the persona's scope and a `role-context-chip`:
   - **Admin** — org-wide: squad breakdown (computed headcount/availability/avg readiness), active alerts, participation-by-program chart. AI insight is org-level (names the squads carrying injury load, computed — no single-athlete callout). Carries a **Federation→Program→Squad hierarchy filter** (`HierarchyFilter`) that re-scopes the whole view in place — every KPI card, the squad breakdown, scoped alerts, the computed participation chart and the AI insight recompute from the drilled roster.
   - **Director** — selection pool (top talent scores), trending up/down, readiness trend (squad vs Arjun). Same **hierarchy drill-down filter** as Admin — KPI cards, selection pool, trending lists and the AI insight all re-scope in place.
   - **Coach** — Sprint A squad: today's session + live attendance counts, squad ACWR bars (1.3 reference line), attention-first roster.
   - **Physio** — return-to-play queue (Arjun first, ordered by attention) with per-case stage progress, wellness flags.
   - **Scientist** — squad-framed: per-athlete ACWR band chart, squad load-vs-readiness scatter (danger quadrant), squad anomaly-detection insight (highest-ACWR cases, not Arjun-centric), highest-mechanical-risk ranking.
   - **Nutritionist** — squad fuelling compliance only (no injury/rehab content): adherence-by-athlete (lowest first, drift detection) + adherence-by-squad chart vs the 80% target.
   - **Ops** — onboarding pipeline (funnel + per-athlete advance), document-verification queue, week's schedule.
   - **Athlete** — single responsive, **athlete-agnostic** self-view (no phone frame): stacks on mobile, flows into a two-column `md:` grid from 768px. Readiness ring, rehab progress + milestone timeline, today's plan, wellness check-in, tappable task list, plus four athlete-facing surfaces: **My weekly classes** (coach-assigned enrolment from `athlete.classes` × `TRAINING_CLASSES`, with an empty state when none assigned), **My diet plan** (the nutritionist's assigned plan from `nutritionPlans[id]` if present, else the derived plan — badged *nutritionist-assigned* vs *auto-derived*, with kcal + carbs/protein/fat + adherence), an **Onboarding status tracker** (invited → registered → docs review → active stepper driven by `athlete.onboarding`), and a **Notifications feed** synthesised from the records that touched the athlete (active AI risk alerts, physio injury-history notes, nutritionist plan assignment, coach class enrolment, ops onboarding state). Renders whichever athlete is signed in (driven by `athleteId` / `me`) and adapts its content — a rehab story for athletes with an active injury vs an availability/training story for fit athletes. Uses the standard dashboard PageHeader.

   **Demo sign-in landing (`/login`):** a credential-free role chooser is the app's entry point — faithful to the spec's exclusion of real auth/accounts, it has **no password fields**. The page opens once per browser session (gated by a `sessionStorage` `ams-entered` flag; `App.js`'s `EntryGate` redirects `/` → `/login` until a choice is made) and offers two tabs: **I'm an athlete** (profile cards from the same dynamic `ATHLETE_LOGINS` + runtime-registered list, plus an **invite-code** field that matches a registered athlete's `inviteCode`/id and signs them in to finish onboarding) and **I'm staff** (one card per non-athlete role with the `STAFF` name + a role blurb). A *Skip — explore as Performance Director* shortcut and the in-app TopBar switcher both remain, so the landing never blocks the demo mid-flow.

   **Login as athlete:** the TopBar account menu lists three curated switchable athlete profiles (`ATHLETE_LOGINS`: Arjun Sharma — deep rehab, Manish Reddy — return-to-play, Rohan Verma — fully available) via `loginAsAthlete(id)`, which sets `athleteId` + `role: "athlete"`. **The list is dynamic:** any athlete registered at runtime (i.e. not in `ATHLETES_SEED`) is appended automatically, labeled by their onboarding stage (`invited — awaiting self-register`, `self-registered — docs pending`, etc.), so a freshly-registered athlete can be signed in and walked through the whole journey. The selected `athleteId` is persisted and threaded through the RBAC layer (`scopeAthletes`/`selfAthleteId` in `lib/access.js`), so every athlete-scoped surface — the self-view, scoped module data and the Copilot snapshot — follows the signed-in athlete.

   **Fresh-athlete management journey:** a newly-registered athlete (`addAthlete` seeds `classes: []`, `nutritionAdherence`, `weight` alongside the existing fields) can be configured by every stakeholder: Ops add + verify, Athlete self-register + doc upload, Physio injury report / RTP, Admin/Director coach assignment, **Nutritionist diet assignment** (item 6) and **Coach class enrolment** (item 3).

   **Per-persona tailoring rule:** Arjun's injury story appears only where it is genuinely relevant to that role — Physio (front and centre), Coach (one alert item) and Director (one of several flagged). Admin, Scientist, Nutritionist and Ops views are framed squad-wide / org-wide and do not centre Arjun.
2. **Athletes** — registry DataTable (scoped: an athlete sees only their own row), search + status filter + **tag filter**, **tags column**. "Add athlete" is gated to `athletes.add` (ops/admin) and enrols at the **invited** stage with a generated `inviteCode`. Click-row drawer with Overview / Training / Medical / Documents tabs; Overview tab manages **athlete tags** (add/remove gated to `athletes.tag`), shows an **onboarding advance** button gated to `athletes.verify`, a **readiness-score breakdown** and a **coach-assignment** control. Documents tab prefers the live-captured `athlete.documents` over seed.

   **Readiness scoring (`lib/readiness.js`):** readiness is a computed 0–100 composite of five weighted daily wellness signals — sleep quality (25%), muscle freshness (20%), HRV/autonomic (25%), energy/fatigue (15%), mood/stress (15%) — minus an ACWR load penalty when load sits outside the 0.8–1.3 sweet spot. Every athlete carries a `wellness` block in the seed authored so `computeReadiness` reproduces its stored readiness exactly; `addAthlete` seeds a `defaultWellness` profile. The detail panel renders the per-factor contribution bars + weights and any load penalty (`readinessBreakdown`).

   **Coach assignment:** the detail panel shows the current coach and, for `athletes.assignCoach` roles (admin/director), a reassign dropdown over `COACHES` backed by the `assignCoach` mutator — reassigning an athlete moves them between coach data scopes (a coach only sees athletes they coach).

   **Onboarding lifecycle (invite → self-register → verify → active):** a single stage-aware `OnboardingDialog` (launched from the Ops dashboard pipeline + verification queue) walks an athlete end to end, backed by three real `AppContext` mutators:
   - `addAthlete` (Ops/Admin) — creates the row at **invited** with an `inviteCode` and a copyable invite link.
   - `selfRegisterAthlete` (Athlete) — captures profile (DOB/phone/emergency contact) + three consents → **pending**.
   - `submitOnboardingDocs` (Athlete) — uploads the four `REQUIRED_DOCS` (federation form, medical clearance, anti-doping consent, photo ID) → **review**.
   - `verifyOnboarding` (Ops/Admin) — signs off the documents → **active**, sets `docsVerified` and marks every document verified.
3. **Training & periodisation** — macro→meso→micro timeline, current microcycle with load bars, AI load-reduction Insight with **Accept** button that swaps Friday/Saturday to recovery (real state change). ACWR 8-week chart with sweet-spot band. Exercise-library session builder. **Session attendance roster** — per-athlete present/late/absent/excused toggles with live count summary (real state via `setAttendanceStatus`). **Class enrolment** (capability `training.assign`, coach/director/admin) — a coach picks a scoped athlete and toggles enrolment across the weekly `TRAINING_CLASSES` (Acceleration / Strength / Tempo / Recovery / Max velocity / Power); enrolment persists on `athlete.classes` via `assignAthleteClasses` and the athlete is auto-added to today's attendance roster so the assignment flows downstream.
4. **Medical & injury (priority)** — fully interactive SVG body map (20 regions), Arjun's right thigh pre-coloured rehab-blue. Detail panel with AI prediction record, **5-stage RTP tracker** (Reported → Under treatment → Rehab → Return-to-play → Cleared) advanced via real state mutation; reaching Cleared turns region green and sets athlete to available. Healthy regions show mild/moderate/severe Report buttons that immediately colour the region. **Clinical timeline** per injury (history entries) with an **add-note** input that appends real entries; stage advances and new reports also append to the timeline. Squad injury log table + a **per-athlete wellness check-ins chart** (`wellnessTrend` in `lib/readiness.js` derives a deterministic 7-day sleep/soreness/mood series from the selected athlete's `wellness` block — athletes carrying load drift worse across the week, fresh athletes stay flat — so the chart follows the body-map athlete selector and respects data scope).
5. **Sports science** — scope-aware **per-athlete monitoring** driven by an athlete selector (`ss-athlete-select`, defaults to `me` then Arjun); every surface follows the selection and respects data scope. Surfaces a **Readiness accent StatCard** (`athlete.readiness` + `readinessVariant`) alongside last-recorded HRV/sleep/ACWR; per-athlete HRV/sleep/load series are derived in `lib/readiness.js` (`hrvSeries`/`sleepSeries`/`loadSeries` — real seed arrays when present, e.g. Arjun, otherwise grounded in the athlete's wellness snapshot/ACWR and stress-modulated). The **AI risk-detection insight** is adaptive: Arjun keeps the 3-signal injury-prediction narrative; other athletes get an elevated-vs-clear summary computed from ACWR>1.3 / readiness<65. Recovery-actions list adapts (rich rehab set for Arjun, load-management set for low-readiness, baseline set otherwise). **GPS external-load card** (ComposedChart: total/HSR/sprint distance bars + max-speed line over last 7 sessions) with a rolling-totals summary panel, both showing a graceful **"no GPS feed" empty-state** for athletes without a vest. **Wearable-integration stub** (connect/disconnect toggles for Catapult/WHOOP/Polar/Oura, local UI only).
6. **Nutrition** — scope-aware **per-athlete** fuelling module driven by an athlete selector (`nutrition-athlete-select`, defaults to `me` then Arjun); every surface follows the selection and respects data scope. The four KPI cards (kcal target, plan adherence, avg hydration, body fat) are **computed from the selected athlete** rather than hardcoded. Diet plan (macro split with kcal-share percentages derived from real grams + meal schedule), 7-day adherence area chart, hydration chart, supplement-compliance list and the 4-week DEXA body-composition trend are all per-athlete: real seed data for Arjun (`NUTRITION_PLAN` + `bodyComp`), otherwise derived in `lib/nutrition.js` (`nutritionPlan`/`bodyCompSeries` — grounded deterministically in the athlete's `nutritionAdherence` scalar and weight). **Diet assignment** (capability `nutrition.edit`) — a nutritionist opens an "Assign diet / Edit plan" dialog to set the prescribed kcal target, macro grams and supplement protocol; the saved plan is stored as an override in `nutritionPlans[athleteId]` (`saveNutritionPlan`) and supersedes the derived plan, while observed adherence/hydration stay as logged. The kcal card and plan header flip to "nutritionist-assigned". Squad-wide compliance ranking lives on the Nutritionist dashboard.
7. **Assessments & TID** — fitness test register (live `fitnessTests` state) with a **test-date column** (`recordedOn`, stamped on new entries and seeded across the cohort), **Record result** dialog that appends a real test row via `addFitnessTest` (benchmark auto-derived from 30 m time), talent ranking computed from live `talentScore`. A **page-level athlete selector** drives the **progression chart** — the selected athlete's composite-score trajectory vs squad average, derived per-athlete in `lib/assessments.js` (`talentProgression` — real `TALENT_PROGRESSION` array for Arjun, otherwise a deterministic ramp to the athlete's current `talentScore`). The 30 m **comparative chart highlights the selected athlete's bar** (brand-blue) against the rest of the scoped squad.
8. **Analytics & BI** — drill-down breadcrumb that **re-scopes the actual cohort** (Federation→Athletics→Sprints→Sprint A), recomputing every stat card, the readiness trend, injury-rate scaling, participation and the AI insight; **Export report** downloads a real CSV of the scoped roster.
9. **AI Copilot** — live streaming chat backed by Claude Sonnet 4.5 (`POST /api/copilot/chat`, SSE token frames via emergentintegrations). The system prompt is rebuilt per request from a **live state snapshot the frontend sends**, so answers are grounded on current data. The snapshot is **role-scoped**: staff get their full scope; an athlete (`me`) gets stats/athletes/injuries filtered to their own record with alerts emptied, plus a self-focused suggestion set and welcome/footer copy — the model can never see other athletes.

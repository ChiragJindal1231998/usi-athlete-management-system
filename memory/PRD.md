## Access control & data scoping (RBAC)
All nine nav tabs stay **visible to every role** ("show all tabs, scope data only" â€” no route guards, no hidden modules). Differentiation happens *inside* each module along two orthogonal dimensions, both centralised in `frontend/src/lib/access.js` and consumed via `AppContext`:

- **Data scope** â€” `scopeAthletes(role, athletes)`: an athlete sees only their own record (`ROLE_IDENTITY.athlete â†’ SPR-014`, Arjun); a coach sees only athletes they coach (`ROLE_IDENTITY.coach â†’ Meera Iyer`, i.e. Sprint A/B); all other staff are federation-wide. Modules read `scopedAthletes` / `me` (the self athlete, or null for staff) rather than the raw `athletes` list, and surface a `<ScopeNote>` banner stating what's in view.
- **Capabilities** â€” `can(role, capability)`: write affordances are gated by an owning-role map (e.g. `injury.report`/`injury.advance` â†’ physio, `training.acceptAI`/`training.moveSession` â†’ coach, `nutrition.edit` â†’ nutritionist, `assessment.record` â†’ scientist/director, `athletes.add`/`athletes.verify` â†’ ops/admin, `analytics.export` â†’ ops/director/admin, `self.*` â†’ athlete). Non-owning roles see the same data **read-only** (buttons hidden or replaced with status badges).
- **Athlete exception (explicit):** the athlete persona never sees the "add athlete" affordance and the Copilot snapshot is scoped to their own record only (`me` â†’ stats/athletes/injuries filtered to self, alerts emptied), so the LLM can never answer about other athletes.
